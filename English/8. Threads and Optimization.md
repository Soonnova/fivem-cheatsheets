# Threads and Performance Optimization

In FiveM, thread management is the number one factor impacting client performance (FPS). A smooth server doesn't just rely on the host machine; it primarily depends on the quality of code written by the developers.

## Understanding the Game Loop

The GTA V engine runs in an infinite loop that refreshes the image approximately 60 times per second (1 frame every 16ms).

### CreateThread: Your Parallel Process
`CreateThread` allows you to launch a `while` loop without "freezing" the rest of the game.

```lua
CreateThread(function()
    while true do
        -- Code executed at every iteration
        print("I'm running!")
        Wait(0) -- YIELDS control back to the game engine.
    end
end)
```

**Why is `Wait(0)` mandatory?**
Without it, your `while` loop would demand 100% of the CPU to run as fast as possible. It wouldn't allow the game time to calculate the next frame, causing an instant crash.

## Optimization through Sleep (Sleep Time)

The golden rule for "0.00ms": **Only make the script work if it is strictly necessary.**

### Example: Interactive Marker Display
We will transform a heavy script into an invisible one.

```lua
local targetPosition = vector3(425.1, -979.5, 30.7)

CreateThread(function()
    while true do
        -- By default, the script "sleeps" for 1.5 seconds. Resmon impact: 0.00ms
        local sleepTime = 1500 
        
        local playerCoords = GetEntityCoords(PlayerPedId())
        local distance = #(playerCoords - targetPosition)

        -- If within 20 meters, increase polling frequency
        if distance < 20.0 then
            sleepTime = 500 -- Checking twice per second
            
            -- If within 3 meters, switch to "real-time" mode
            if distance < 3.0 then
                sleepTime = 0 -- Necessary so the marker doesn't flicker
                DrawMarker(1, targetPosition, 0, 0, 0, 0, 0, 0, 1.0, 1.0, 1.0, 255, 0, 0, 100)
            end
        end

        Wait(sleepTime)
    end
end)
```

## Profiling: Reading the Resmon (F8 -> resmon 1)

The `resmon` displays the CPU consumption in milliseconds (ms).

| Value | Pedagogical Diagnosis |
| :--- | :--- |
| **0.01 - 0.04 ms** | Excellent. The script is passive or perfectly optimized. |
| **0.10 - 0.30 ms** | Average. Too many active loops or heavy natives. |
| **> 0.80 ms** | Critical. This script alone can lower players' FPS. |

## Expert Tips for 0.00ms

1.  **Cache IDs**: Call `PlayerPedId()` once at the start of your loop if you use it multiple times.
2.  **Declare outside loops**: Don't create new tables `{}` inside a `Wait(0)`. Lua's Garbage Collector will have to clean them up later, which can cause micro-stutters.
3.  **State Filtering**: If your player is dead or in an inventory, set your movement or marker loops to `Wait(1000)`.

```lua
local isMenuOpen = false
-- Listen for state change via an event
RegisterNetEvent("menu:state", function(open) isMenuOpen = open end)

CreateThread(function()
    while true do
        if isMenuOpen then
            Wait(1000) -- Do NOTHING while the menu is open
        else
            -- Normal logic
            Wait(0)
        end
    end
end)
```

> [!CAUTION]
> **Heavy Natives**: Certain natives like `GetActivePlayers()` or `GetClosestVehicle()` are very resource-intensive. NEVER use them in a `Wait(0)` without a distance condition first.
