# Server and Client Communication

FiveM operates on a distributed architecture. Understanding how to exchange data between the client (the player's game instance) and the server (the central brain) is the foundation of all multiplayer scripting.

## Why separate Client and Server?

Imagine a game where your computer alone decided how much money you had. You could simply modify a file on your PC to become a billionaire.

This is why we split logic:
*   **The Client** handles what the player **sees** and **does** (driving, shooting, opening a menu).
*   **The Server** handles what the player **owns** and the **rules** (money, inventory, database).

## Events: The Global Network

To communicate, we use "Events." Think of it as making a phone call between scopes.

### 1. Client to Server (TriggerServerEvent)
The player asks the server to perform a critical action.

**Client (Lua)**:
```lua
-- "Server, I would like to buy some bread."
TriggerServerEvent("script:buyBread", 1)
```

**Server (Lua)**:
```lua
RegisterNetEvent("script:buyBread", function(quantity)
    -- 'source' is the automatic ID of the player who called the event
    local _source = source 
    
    -- SECURITY: The server MUST verify if the player has enough money
    -- If we did this on the client, a cheater could skip this check
    print("Player " .. _source .. " wants to buy " .. quantity .. " bread(s)")
end)
```

### 2. Server to Client (TriggerClientEvent)
The server informs one player (or all of them) about a change.

**Server (Lua)**:
```lua
-- Send to a specific player via their ID
TriggerClientEvent("script:notify", targetPlayerId, "Purchase successful!")

-- Send to everyone (using -1)
TriggerClientEvent("script:notify", -1, "The server will restart in 5 minutes")
```

**Client (Lua)**:
```lua
RegisterNetEvent("script:notify", function(message)
    -- Display the notification received from the server
    exports['my_hud']:displayNotify(message)
end)
```

## Framework Integration

Frameworks make player management easier by transforming the raw `source` ID into a rich object.

### ESX Example (Server Side)
```lua
RegisterNetEvent("script:giveMoney", function()
    -- Convert 'source' ID into an xPlayer object
    local xPlayer = ESX.GetPlayerFromId(source)
    
    -- Now we can use simple built-in functions
    xPlayer.addAccountMoney('bank', 500)
end)
```

### QB-Core Example (Server Side)
```lua
RegisterNetEvent("script:checkJob", function()
    local Player = QBCore.Functions.GetPlayer(source)
    
    if Player.PlayerData.job.name == "police" then
        -- Action authorized
    end
end)
```

## Latent Events: For Large Data

If you send a massive table (e.g., 1000 vehicles) via a standard `TriggerClientEvent`, it can cause a "lag spike." Use `TriggerLatentClientEvent` to send data in small chunks.

```lua
-- Server to Client
-- 50000 = transfer speed in bytes per second
TriggerLatentClientEvent("script:receiveHugeTable", targetId, 50000, myGargantuanTable)
```

> [!CAUTION]
> **Source is Sacred**: Inside a server-side `RegisterNetEvent`, ALWAYS use `source` (or a local `_source = source`) to identify who is speaking. Never let the client send its own ID as an argument, as a cheater could impersonate another player (like an admin).
