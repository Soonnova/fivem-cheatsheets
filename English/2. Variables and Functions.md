# Variables and Functions

This guide dives deep into memory management and logic organization. By understanding how Lua handles these two pillars, you will transition from "copy-pasting" code to truly mastering your FiveM scripts.

## Understanding Scope

In Lua, "scope" defines where a variable is visible and usable. Poor scope management is the leading cause of performance issues and data leaks.

### Why always prefer "local"?

As seen previously, a local variable is confined to its block (function, file, or loop).

*   **Memory Efficiency**: As soon as execution leaves the block (e.g., when a function finishes), Lua destroys the local variables, freeing up RAM.
*   **Speed**: Lua doesn't need to search through the global environment table `_G` (which contains thousands of entries) to find your variable.

```lua
-- LOCAL ACCESS (Fast and Safe)
local playerName = "Citizen" 

-- GLOBAL ACCESS (Slow and Risky)
-- Without the local keyword, Lua searches everywhere to see if 'pMoney' already exists.
pMoney = 5000 
```

## Anatomy of a Function

A function is like a "machine": it takes input data (arguments), performs work, and returns a result.

### Declaration and Return
```lua
local function calculateTax(amount)
    local taxRate = 0.20
    -- The function 'stops' at the return keyword and yields the result
    return amount * taxRate
end

-- We store the result of the function's work in a variable
local tax = calculateTax(100) -- 'tax' will be 20
```

### Anonymous Functions (Callbacks)
In FiveM, you will frequently encounter functions without names. They are used as "instructions to be executed later."

```lua
-- We provide a function as an argument to CreateThread
CreateThread(function()
    print("This code runs in its own separate process")
end)
```

## Framework Integration (ESX & QB-Core)

Frameworks use **objects** (complex tables) to centralize player data.

### ESX Example: The xPlayer Object
ESX transforms the "source" (player ID) into a rich object with built-in functions.

```lua
-- Server Side
local xPlayer = ESX.GetPlayerFromId(source)

if xPlayer then
    -- Using a function belonging to the xPlayer object
    local money = xPlayer.getMoney()
    xPlayer.addMoney(100) -- Action: give $100
end
```

### QB-Core Example: Player Data
QB-Core functions similarly but structures its data via nested tables.

```lua
-- Client Side
local PlayerData = QBCore.Functions.GetPlayerData()

-- Accessing info via nested tables
local jobLabel = PlayerData.job.label
local isDead = PlayerData.metadata["isdead"]
```

## Advanced Concepts: Closures

A "closure" is a function that "captures" the variables surrounding it at the time of its creation. This is useful for creating private state or counters.

```lua
local function createCounter()
    local count = 0 -- This variable is "remembered" by the inner function
    return function()
        count = count + 1
        return count
    end
end

local myCounter = createCounter()
print(myCounter()) -- Output: 1
print(myCounter()) -- Output: 2
```

> [!TIP]
> **Golden Rule**: If your function is longer than 50 lines, it is likely too complex. Break it down into smaller, specialized functions. "Deconstructed" code is ten times easier to debug.
