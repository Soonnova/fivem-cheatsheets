# Threads et Optimisation des Performances

Sur FiveM, la gestion des threads est le facteur numéro 1 impactant les performances client (FPS). Un serveur fluide ne dépend pas seulement de la machine hôte, mais surtout de la qualité du code écrit par les développeurs.

## Comprendre le Thread Principal (Game Loop)

Le moteur de GTA V s'exécute dans une boucle infinie qui rafraîchit l'image environ 60 fois par seconde (soit 1 frame toutes les 16ms).

### CreateThread : Votre processus parallèle
`CreateThread` permet de lancer une boucle `while` sans "freezer" le reste du jeu. 

```lua
CreateThread(function()
    while true do
        -- Code exécuté à chaque itération
        print("Je tourne !")
        Wait(0) -- CÈDE le contrôle au moteur de jeu.
    end
end)
```

**Pourquoi le `Wait(0)` est obligatoire ?**
Sans lui, votre boucle `while` demanderait 100% du processeur pour s'exécuter le plus vite possible. Elle ne laisserait pas le temps au jeu de calculer l'image suivante, provoquant un crash instantané.

## L'Optimisation par le Sommeil (Sleep Time)

La règle d'or pour le "0.00ms" : **Ne faites travailler le script que si c'est strictement nécessaire.**

### Exemple : Affichage d'un Marker interactif
Nous allons passer d'un script qui consomme beaucoup à un script invisible.

```lua
local positionCible = vector3(425.1, -979.5, 30.7)

CreateThread(function()
    while true do
        -- Par défaut, le script "dort" 1.5 seconde. Impact resmon : 0.00ms
        local tempsAttente = 1500 
        
        local playerCoords = GetEntityCoords(PlayerPedId())
        local distance = #(playerCoords - positionCible)

        -- Si on est à moins de 20 mètres, on accélère la fréquence
        if distance < 20.0 then
            tempsAttente = 500 -- On vérifie 2 fois par seconde
            
            -- Si on est à moins de 3 mètres, on passe en mode "temps réel"
            if distance < 3.0 then
                tempsAttente = 0 -- Nécessaire pour que le marker ne clignote pas
                DrawMarker(1, positionCible, 0, 0, 0, 0, 0, 0, 1.0, 1.0, 1.0, 255, 0, 0, 100)
            end
        end

        Wait(tempsAttente)
    end
end)
```

## Profiling : Lire le Resmon (F8 -> resmon 1)

Le `resmon` affiche la consommation en millisecondes (ms).

| Valeur | Diagnostic Pédagogique |
| :--- | :--- |
| **0.01 - 0.04 ms** | Excellent. Le script est passif ou parfaitement optimisé. |
| **0.10 - 0.30 ms** | Moyen. Trop de boucles actives ou de natives lourdes. |
| **> 0.80 ms** | Critique. Ce script à lui seul peut faire baisser les FPS des joueurs. |

## Astuces d'Expert pour le 0.00ms

1.  **Cachez les IDs** : Appelez `PlayerPedId()` une seule fois au début de votre boucle si vous l'utilisez plusieurs fois.
2.  **Sortez les variables** : Ne créez pas de nouvelles tables `{}` à l'intérieur d'un `Wait(0)`. Lua doit les "nettoyer" plus tard (Garbage Collector), ce qui crée des micro-stutters.
3.  **Filtrage par État** : Si votre joueur est mort ou dans un inventaire, mettez vos boucles de mouvement ou de marker en `Wait(1000)`.

```lua
local estDansMenu = false
-- On écoute un changement d'état via un événement
RegisterNetEvent("menu:etat", function(ouvert) estDansMenu = ouvert end)

CreateThread(function()
    while true do
        if estDansMenu then
            Wait(1000) -- On ne fait RIEN tant que le menu est ouvert
        else
            -- Logique normale
            Wait(0)
        end
    end
end)
```

> [!CAUTION]
> **Natives Lourdes** : Certaines natives comme `GetActivePlayers()` ou `GetClosestVehicle()` sont très gourmandes. Ne les utilisez JAMAIS dans un `Wait(0)` sans une condition de distance préalable.
