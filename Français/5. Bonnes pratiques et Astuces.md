# Bonnes Pratiques et Optimisation

Développer sur FiveM ne se limite pas à faire fonctionner un script. Il s'agit de le faire fonctionner **sans impacter les autres**. Un serveur peut avoir 100 scripts ; si chacun consomme un peu trop, le jeu devient injouable (perte de FPS).

## La Lisibilité : Écrire pour les humains

Le code est lu beaucoup plus souvent qu'il n'est écrit. Un code propre facilite le débogage et la collaboration.

### Standards de Nommage
Choisissez une convention et tenez-vous-y. La clarté prime sur la brièveté.

| Élément | Recommandation | Pourquoi ? |
| :--- | :--- | :--- |
| **Variables** | `argentJoueur` ou `playerMoney` | Évitez `aj` ou `m`. On doit savoir ce qu'il y a dedans sans relire tout le script. |
| **Fonctions** | `verifierPermis()` | Utilisez des verbes d'action. |
| **Événements** | `script:client:ouvrirMenu` | Utilisez des préfixes pour savoir quel script a déclenché l'événement. |

## Modularité : Ne vous répétez pas (DRY)

Le principe **DRY** (*Don't Repeat Yourself*) consiste à ne pas copier-coller le même code à plusieurs endroits. Utilisez des `exports` pour partager des fonctions entre vos scripts.

```lua
-- Dans le script 'utils' (fxmanifest : exports { 'maFonction' })
function maFonction(texte)
    return string.upper(texte)
end

-- Dans un autre script
local texteMaj = exports['utils']:maFonction("test")
```

## Optimisation Technique (Le Resmon)

Le `resmon` (Resource Monitor) est l'outil qui mesure le temps processeur utilisé par votre script en millisecondes (ms).

### Localisation des Natives
Accéder à une fonction globale de FiveM (Native) est plus lent qu'accéder à une fonction locale. Dans une boucle intensive, **localisez vos natives**.

```lua
-- Haut du fichier
local GetEntityCoords = GetEntityCoords
local PlayerPedId = PlayerPedId

-- La boucle gagnera quelques micro-secondes précieuses à chaque itération
while true do
    local coords = GetEntityCoords(PlayerPedId())
    Wait(0)
end
```

## Sécurité : Le Client est un menteur

C'est la règle d'or de tout développeur multijoueur. **Tout ce qui vient du client peut être falsifié par un logiciel de triche.**

### Le bon réflexe
Le serveur doit toujours être le seul juge pour les actions importantes (argent, inventaire, permissions).

```lua
-- Côté Serveur (SÉCURISÉ)
RegisterNetEvent("shop:buyItem", function(itemName)
    local _source = source
    local xPlayer = ESX.GetPlayerFromId(_source)
    local prix = 500 -- Le prix est défini sur le SERVEUR, pas envoyé par le client
    
    if xPlayer.getMoney() >= prix then
        xPlayer.removeMoney(prix)
        xPlayer.addInventoryItem(itemName, 1)
    end
end)
```

## Gestion des Erreurs : pcall

Si une partie de votre code risque de "crasher" (par exemple si vous lisez une table qui pourrait être vide), utilisez `pcall` (*protected call*).

```lua
local succes, resultat = pcall(function()
    -- Code risqué ici
    return maTableCible[1].nom
end)

if not succes then
    print("Oups, une erreur a été évitée : " .. resultat)
end
```

> [!TIP]
> **Le secret du 0.00ms** : Un script parfait est un script qui "dort" (Wait élevé) quand le joueur n'est pas en train d'interagir spécifiquement avec lui. L'optimisation, c'est l'art de ne rien faire par défaut.
