# Communication Client / Serveur

FiveM repose sur une architecture distribuée. Comprendre comment échanger des données entre le client (le jeu du joueur) et le serveur (le cerveau central) est la base de tout script multijoueur.

## Pourquoi séparer le Client et le Serveur ?

Imaginez un jeu où votre ordinateur décide seul de combien d'argent vous avez. Il suffirait de modifier un petit fichier sur votre PC pour devenir milliardaire. 

C'est pour cela que :
*   **Le Client** s'occupe de ce que le joueur **voit** et **fait** (conduire, tirer, ouvrir un menu).
*   **Le Serveur** s'occupe de ce que le joueur **possède** et des **règles** (argent, inventaire, base de données).

## Les Événements (Events) : Le téléphone du réseau

Pour communiquer, on utilise des "Events". C'est comme passer un coup de téléphone.

### 1. Du Client vers le Serveur (TriggerServerEvent)
Le joueur demande au serveur d'effectuer une action importante.

**Client (Lua)** :
```lua
-- "Monsieur le Serveur, je voudrais acheter du pain."
TriggerServerEvent("script:acheterPain", 1)
```

**Serveur (Lua)** :
```lua
RegisterNetEvent("script:acheterPain", function(quantite)
    -- 'source' est l'ID automatique du joueur qui a appelé
    local _source = source 
    
    -- SÉCURITÉ : Le serveur vérifie si le joueur a l'argent
    -- Si on faisait ça côté client, un tricheur pourrait sauter cette vérification
    print("Le joueur " .. _source .. " veut acheter " .. quantite .. " pain(s)")
end)
```

### 2. Du Serveur vers le Client (TriggerClientEvent)
Le serveur informe un joueur (ou tous) d'un changement.

**Serveur (Lua)** :
```lua
-- Envoyer à un joueur précis via son ID
TriggerClientEvent("script:notif", idDuJoueur, "Achat réussi !")

-- Envoyer à tout le monde (ID -1)
TriggerClientEvent("script:notif", -1, "Le serveur redémarre dans 5 minutes")
```

**Client (Lua)** :
```lua
RegisterNetEvent("script:notif", function(message)
    -- On affiche la notification reçue du serveur
    exports['mon_hud']:displayNotify(message)
end)
```

## Intégration aux Frameworks

Les frameworks facilitent grandement la gestion des joueurs en transformant la simple `source` en un objet complet.

### Exemple ESX (Serveur)
```lua
RegisterNetEvent("script:donnerArgent", function()
    -- On convertit l'ID 'source' en objet xPlayer
    local xPlayer = ESX.GetPlayerFromId(source)
    
    -- Maintenant on peut utiliser des fonctions simples
    xPlayer.addAccountMoney('bank', 500)
end)
```

### Exemple QB-Core (Serveur)
```lua
RegisterNetEvent("script:verifierJob", function()
    local Player = QBCore.Functions.GetPlayer(source)
    
    if Player.PlayerData.job.name == "police" then
        -- Action autorisée
    end
end)
```

## Événements Latents : Pour les grosses données

Si vous envoyez une table énorme (ex: 1000 véhicules) via un `TriggerClientEvent` classique, cela peut provoquer un "lag réseau" (lag spike). Utilisez `TriggerLatentClientEvent` pour envoyer les données petit à petit.

```lua
-- Serveur vers Client
-- 50000 = vitesse d'envoi en octets par seconde
TriggerLatentClientEvent("script:reçevoirGrosseTable", idJoueur, 50000, maTableGeante)
```

> [!CAUTION]
> **La Source est sacrée** : Dans un `RegisterNetEvent` serveur, utilisez TOUJOURS `source` (ou une variable locale `_source = source`) pour savoir qui parle. Ne laissez jamais le client envoyer son propre ID en argument, car un tricheur pourrait se faire passer pour un autre joueur (admin, par exemple).
