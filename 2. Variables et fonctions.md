# Variables et Fonctions

Ce guide approfondit la gestion de la mémoire et l'organisation de la logique. En comprenant comment Lua gère ces deux piliers, vous passerez d'un "copier-coller" de code à une réelle maîtrise de vos scripts FiveM.

## Comprendre la Portée (Scope)

En Lua, la "portée" définit où une variable est visible et utilisable. Une mauvaise gestion de la portée est la première cause de bugs de performance et de "fuite de données".

### Pourquoi toujours privilégier "local" ?

Comme vu précédemment, une variable locale est confinée à son bloc (fonction, fichier, boucle). 

*   **Libération de mémoire** : Dès que l'exécution sort du bloc (par exemple quand une fonction se termine), Lua détruit les variables locales, libérant de la RAM.
*   **Rapidité** : Lua n'a pas besoin de chercher dans la table globale `_G` (qui contient des milliers d'entrées) pour trouver votre variable.

```lua
-- ACCÈS LOCAL (Rapide et sûr)
local nomJoueur = "Citizen" 

-- ACCÈS GLOBAL (Lent et risqué)
-- Sans le mot-clé local, Lua cherche partout sur le serveur si 'pMoney' existe déjà.
pMoney = 5000 
```

## Structure d'une Fonction

Une fonction est une "machine" : elle prend des données en entrée (arguments), effectue un travail, et renvoie un résultat (retour).

### Déclaration et Retour
```lua
local function calculerTVA(montant)
    local taux = 0.20
    -- La fonction 's'arrête' au mot-clé return et donne le résultat
    return montant * taux
end

-- On stocke le résultat du travail de la fonction dans une variable
local taxe = calculerTVA(100) -- taxe vaudra 20
```

### Fonctions Anonymes (Callbacks)
Dans FiveM, vous rencontrerez souvent des fonctions sans nom. Elles sont utilisées comme des instructions "à exécuter plus tard".

```lua
-- On donne une fonction en argument à CreateThread
CreateThread(function()
    print("Ce code s'exécute dans son propre petit processus")
end)
```

## Intégration aux Frameworks (ESX & QB-Core)

Les frameworks utilisent des **objets** (des tables complexes) pour centraliser les données d'un joueur.

### Exemple ESX : L'objet xPlayer
ESX transforme la "source" (l'ID du joueur) en un objet riche en fonctions.

```lua
-- Côté Serveur
local xPlayer = ESX.GetPlayerFromId(source)

if xPlayer then
    -- On utilise une fonction de l'objet xPlayer
    local argent = xPlayer.getMoney()
    xPlayer.addMoney(100) -- Action : donner 100$
end
```

### Exemple QB-Core : Le Player Data
QB-Core fonctionne de manière similaire mais structure ses données différemment.

```lua
-- Côté Client
local PlayerData = QBCore.Functions.GetPlayerData()

-- On accède aux informations via des tables imbriquées
local jobLabel = PlayerData.job.label
local estMort = PlayerData.metadata["isdead"]
```

## Concepts Avancés : Les Closures

Une "closure" est une fonction qui "capture" les variables qui l'entourent au moment de sa création. C'est très utile pour créer des compteurs ou des systèmes d'états privés.

```lua
local function creerCompteur()
    local compte = 0 -- Cette variable est "mémorisée" par la fonction interne
    return function()
        compte = compte + 1
        return compte
    end
end

local monCompteur = creerCompteur()
print(monCompteur()) -- Sortie : 1
print(monCompteur()) -- Sortie : 2
```

> [!TIP]
> **Règle d'or** : Si votre fonction fait plus de 50 lignes, elle est probablement trop complexe. Divisez-la en petites fonctions spécialisées. Un code "découpé" est 10 fois plus facile à déboguer.
